# Архитектура

## Коротко

Локальная папка памяти Claude превращается в **симлинк** (на *nix) или **Junction** (на Windows), указывающий на рабочую копию приватного git-репо. Любая запись Claude в эту папку физически попадает в рабочую копию репо. Дальше обычный git — `push` на GitHub, `pull` на других машинах разносит изменения.

```
┌─────────────┐   git push     ┌─────────────┐   git pull (auto)   ┌─────────────┐
│    Mac      │ ─────────────▶ │   GitHub    │ ──────────────────▶ │     VPS     │
│ (симлинк)   │                │ (приватный  │                     │ (симлинк)   │
└─────────────┘                │  memory     │                     └─────────────┘
                               │   репо)     │
                               └─────────────┘
                                      ▲
                                      │ git pull (auto)
                                      │
                               ┌─────────────┐
                               │   Windows   │
                               │ (Junction)  │
                               └─────────────┘
```

Smart auto-pull раз в 15 минут делает механизмом каждой ОС:
- **macOS** — LaunchAgent (`~/Library/LaunchAgents/com.claude-memory.autopull.plist`) запускает wrapper `~/.local/bin/claude-memory-autopull.sh`.
- **Linux** — cron `*/15 * * * * ~/.local/bin/claude-memory-autopull.sh`.
- **Windows** — Task Scheduler (`ClaudeMemoryAutoPull`) запускает `%LOCALAPPDATA%\ClaudeMemorySmartPull.ps1`.

Wrapper сначала проверяет состояние репо. Он пропускает pull, если есть локальные изменения, уже идёт другой git-процесс или локальная ветка ушла вперёд. На Windows он также пропускает запуск во время игры, VR или высокой нагрузки CPU.

## Почему симлинк а не копирование

Альтернативный подход — держать память в обычной папке, а синхронизацию делать через «копирующий» скрипт. Проблемы:

- **Гонки записи.** Claude пишет в момент, когда скрипт копирует — часть файла уезжает в git, часть остаётся не залитой. Редко, но бывает.
- **Надо помнить запустить.** «Копировать по расписанию» требует того же cron — получается такая же сложность, только с copy вместо git pull плюс больше точек отказа.
- **Mtime сюрпризы.** При копировании `mtime` исходного файла теряется — конфликты разрешать труднее.

Симлинк снимает все три вопроса: Claude пишет **прямо** в git-репо, без копирования.

## Почему Junction а не symlink на Windows

Windows поддерживает два типа «папочных ссылок»:

| Тип | Команда | Нужны ли admin-права | Видимость из ОС | Работает? |
|---|---|---|---|---|
| **Junction** | `mklink /J` | Нет | Как обычная папка | Да |
| Symbolic Link | `mklink /D` | **Да** (или Developer Mode) | Как обычная папка | Да |

Junction работает **без admin-прав**, потому что это reparse point низкого уровня — NTFS-фича. Symbolic Link требует привилегию `SeCreateSymbolicLinkPrivilege`, которая по умолчанию только у Administrators. Для пользовательского скрипта Junction — правильный выбор.

Разница для Claude Code незаметна: обе формы работают как папка.

## Почему auto-pull а не webhook

Webhook от GitHub (push → уведомить все машины → pull) был бы быстрее, но:

- Требует **публичный endpoint** на каждой машине (или прокси). Для Mac и Windows за NAT — нетривиально.
- Требует **секрет** для валидации, его надо хранить безопасно.
- Если машина офлайн — webhook потеряется, pull всё равно нужен при включении.

Cron, launchd или Task Scheduler раз в 15 минут покрывают 99% случаев с нулевой инфраструктурой. Если машина офлайн — при следующем включении первый же pull догонит все изменения.

Задержка 15 минут нормальна: память Claude — это не чат, не real-time. Если очень срочно — `git pull` руками.

## Бэкап и rollback

Установка — деструктивная операция: локальная папка памяти удаляется, на её место ставится симлинк. Скрипты **перед любой деструкцией** делают полный бэкап папки в `$HOME/claude-memory-backup-<timestamp>`.

`rollback.sh` / `rollback.ps1`:
1. Останавливает auto-pull (удаляет cron-запись / LaunchAgent / Task Scheduler задачу).
2. Удаляет симлинк.
3. Копирует бэкап обратно на место.

Git-репо не трогается — его можно оставить для следующей попытки.

## Идемпотентность

Скрипты идемпотентны — повторный запуск:
- Если симлинк уже правильный → сообщают «ок» и выходят.
- Если симлинк сломан или ведёт в другое место → пересоздают.
- Если auto-pull уже настроен → перезаписывают свою задачу.

Запустить можно хоть сто раз подряд — ничего не сломает.

## Что хранить в репо с памятью, а что НЕ хранить

**Хранить:**
- `MEMORY.md` — индекс
- `project_<name>.md`, `feedback_<topic>.md`, `user_*.md`, `reference_<system>.md` — обычные записи
- `session_state.md` — выжимка последней сессии
- `CLAUDE.md` (опционально — чтобы все машины видели одни глобальные инструкции)
- `archive/YYYY-MM.md` — устаревшие записи

**НЕ хранить:**
- **Секреты** (API-ключи, токены, пароли). Держи их в отдельном файле вне git (`~/.claude/secrets/api-keys.env`).
- **Большие бинарники** (PDF, изображения). Git плохо их версионит. Если нужны — в git LFS или отдельный файл-стор.
- **Ephemeral state** — «сейчас я работаю над X». Это живёт в рамках сессии, не нужно в долгой памяти.

## Ограничения

- **Шифрование at-rest.** Репо приватный, но GitHub видит содержимое. Если в памяти реально чувствительные вещи — используй self-hosted git (Gitea) + шифрование диска на VPS.
- **Нет conflict resolution в UI.** Если одновременно пишешь на двух машинах и обе пытаются push — обычный git-конфликт. Разрешать через `git pull --rebase` + ручное разрешение конфликтов.
- **Работает только для Claude Code CLI.** Веб-версия Claude.ai хранит память у Anthropic и про эти файлы ничего не знает.
