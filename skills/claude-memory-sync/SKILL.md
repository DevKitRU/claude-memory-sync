# Claude Memory Sync — повседневная работа

Этот скил — для регулярной работы с git-репо, в который замаплена папка памяти Claude. Установку/переустановку делает `/setup-memory-sync` — здесь только рутина.

## Команды

| Что хочу | Что делает скил |
|---|---|
| Сохранить новую память на GitHub | `save` — git add -A + commit + push |
| Подтянуть свежее с GitHub вручную | `pull` — git pull --rebase с проверкой конфликтов |
| Проверить что не запушено | `status` — git status + git log origin..HEAD |
| Разрешить merge-конфликт | `resolve` — показать файлы, стратегию разрешения |

## Как определить memory-репо

Скил определяет путь так:

1. Смотрит куда ведёт симлинк `~/.claude/projects/<hash>/memory` (на Windows — Junction: `Get-Item -Force ... | .Target`).
2. Поднимается на уровень выше (до корня репо).
3. Проверяет что там есть `.git/`.

Если симлинк не настроен — предложить пользователю запустить `/setup-memory-sync`.

## Инструкции для Claude

### `save` — сохранить память в GitHub

**Когда вызывать:** пользователь сказал «сохрани память», «сохрани», «save memory», «push memory», «запушь память».

**Шаги:**

1. Найти путь к memory-репо (см. выше).
2. `cd <repo> && git status --short` — посмотреть что изменилось.
3. Если чисто — сказать «нечего сохранять, всё уже в git» и выйти.
4. Иначе показать список изменённых файлов и предложить commit message на основе содержания. Например если изменились `project_X.md` и `MEMORY.md` — сообщение: `memory: update project_X + index`.
5. **Перед коммитом** — проверить что pre-commit hook установлен (`.git/hooks/pre-commit`). Если нет — предупредить:
   ```
   ⚠ pre-commit hook не установлен. Риск утечки API-ключей в git.
   Поставить: cp <claude-memory-sync>/hooks/pre-commit .git/hooks/ && chmod +x .git/hooks/pre-commit
   ```
6. `git add -A && git commit -m '<message>'` — если hook откажет (нашёл секрет), **НЕ** использовать `--no-verify`. Показать пользователю что нашёл hook и попросить разобраться.
7. `git push` — если push отклонён (remote ahead) — автоматически сделать `git pull --rebase && git push`, но если в rebase конфликт — перейти в `resolve`.

### `pull` — подтянуть свежее

**Когда вызывать:** «подтяни память», «pull memory», «что там с VPS/Mac», «обнови память», «свежая память».

**Шаги:**

1. Найти memory-репо.
2. `cd <repo> && git fetch --quiet`
3. `git rev-list --count HEAD..@{u}` — сколько коммитов на удалёнке ещё не у нас.
4. Если 0 — «уже up-to-date».
5. Если >0 — `git pull --rebase`.
6. Если conflict — перейти в `resolve`.
7. Если ок — показать `git log --oneline HEAD@{1}..HEAD` — что подтянулось (самые свежие коммиты).

### `status` — что в памяти не запушено

**Когда вызывать:** «что с памятью», «синхронизация памяти», «status memory», «всё ли запушено».

**Шаги:**

1. Найти memory-репо.
2. `cd <repo> && git fetch --quiet 2>/dev/null || true` (не ломаться если офлайн).
3. Вывести кратко:
   - Uncommitted: `git status --porcelain | wc -l`
   - Ahead/behind: `git rev-list --left-right --count @{u}...HEAD`
   - Последний коммит: `git log -1 --format='%h %s (%ar)'`
4. Если есть uncommitted **или** ahead > 0 — посоветовать `save`.
5. Если behind > 0 — посоветовать `pull`.
6. Если и то, и другое — diverged, нужен `pull` сначала (`--rebase`).

### `resolve` — разрешить конфликт

**Когда вызывать:** `git pull --rebase` или `git push` упал с конфликтом.

**Шаги:**

1. `git status --porcelain` — список файлов с `UU`/`AA`/`DD`.
2. Для каждого conflicted-файла показать `git diff <file>` и объяснить пользователю ЧТО конфликтует. Обычно это одна и та же запись `MEMORY.md` или `session_state.md` которую редактировали с двух машин.
3. **Стратегия разрешения:**
   - Если это `session_state.md` — почти всегда взять **наше** (`git checkout --ours session_state.md`), потому что это снапшот текущей сессии.
   - Если это `MEMORY.md` — обычно **смержить руками**: обе стороны добавили разные записи, их нужно объединить.
   - Если это отдельный `project_*.md` / `feedback_*.md` — показать diff, спросить что оставить.
4. После разрешения: `git add <files>`, потом `git rebase --continue` (если был rebase) или `git commit` (если merge).
5. `git push` — проверить что прошло.

## Частые ошибки

- **Не используй `--no-verify`** даже если pre-commit hook откажет. Это обход защиты от утечки токенов. Либо пользователь разберётся с ложным срабатыванием и добавит маркер `SECRETS_OK_PLACEHOLDER`, либо это реальная утечка и её надо убрать.
- **Не используй `git push --force`** в memory-репо. Другие машины с auto-pull могут потерять свои недопушенные коммиты.
- **Не делай `git reset --hard` без явного подтверждения** — удалит локальные правки памяти.
- **Не комитти `session_state.md` бездумно** — если session_state часто меняется между машинами, конфликты будут постоянно. Можно добавить его в `.gitignore` memory-репо если он только локальный.

## Когда сам пользователь должен решить

- Merge-конфликт в `MEMORY.md` — Claude может показать обе версии, но окончательное решение за человеком.
- Push был отклонён потому что чужой коллаборатор запушил — тот же вопрос.
- Pre-commit hook нашёл секрет — только пользователь знает это ложное срабатывание или реальный ключ.
