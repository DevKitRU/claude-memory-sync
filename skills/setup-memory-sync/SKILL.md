# Setup Memory Sync

Claude Code хранит память в локальной папке на каждой машине — между устройствами ничего не синхронизируется. Этот скил делает из локальной папки памяти **симлинк на git-репо пользователя** и настраивает auto-pull, чтобы все машины видели одну и ту же память через GitHub.

## Архитектура после установки

```
Mac    ↘
Linux   → GitHub (приватный claude-memory репо) ← git pull каждые 5 мин
Windows↗
```

Все три машины пишут память напрямую в один git-репо, `git push` → другие машины подхватывают.

## Что делает скил

1. Определяет платформу: `uname` на *nix, `$env:OS` на Windows.
2. Находит git-репо с памятью пользователя. Дефолтные места:
   - macOS: `~/Documents/claude-memory`, `~/claude-memory`
   - Linux: `~/claude-memory`, `~/Documents/claude-memory`
   - Windows: `E:\projects\claude-memory`, `%USERPROFILE%\claude-memory`
3. Запускает соответствующий скрипт из `setup/`:
   - macOS → `setup/mac.sh`
   - Linux → `setup/linux.sh`
   - Windows → `setup/windows.ps1`
4. После установки запускает health-check.
5. В случае ошибки предлагает rollback.

Скрипты идемпотентны — повторный запуск не ломает.

## Что произойдёт с локальной памятью

Скрипт перед любой деструктивной операцией:
1. Делает бэкап локальной папки памяти в домашнюю директорию (например `~/claude-memory-backup-<timestamp>`).
2. Мержит уникальные локальные файлы в git-репо (чтобы не потерять заметки).
3. Заменяет локальную папку на симлинк (или Junction на Windows) к git-репо.
4. Настраивает auto-pull: cron на Mac/Linux, Task Scheduler на Windows.
5. Тестирует запись через симлинк.

## Инструкции для Claude

**Шаг 1 — соберите контекст:**
- Определи платформу через `uname -s` (Darwin/Linux) или через проверку `$env:OS -eq "Windows_NT"`.
- Спроси у пользователя путь к git-репо с памятью или найди его по дефолтным путям.
- Если репо нет — подскажи клонировать:

  ```bash
  # Пользователь клонирует свой приватный репо с памятью:
  git clone https://github.com/<USER>/<MEMORY_REPO>.git ~/Documents/claude-memory
  ```

  Публичный шаблон этого скила и setup-скриптов — `github.com/DevKitRU/claude-memory-sync`. Приватный репо с самой памятью каждый пользователь заводит свой.

**Шаг 2 — покажи план и подтверди:**
Прежде чем запускать, скажи пользователю что произойдёт, включая путь для бэкапа. Это деструктивная операция (хотя и с бэкапом). Подтверди «продолжаем?» если Claude работает не в полностью автономном режиме.

**Шаг 3 — запусти скрипт:**

```bash
# macOS
cd ~/Documents/claude-memory && ./setup/mac.sh

# Linux (VPS)
cd ~/claude-memory && ./setup/linux.sh

# Windows (PowerShell)
cd E:\projects\claude-memory; .\setup\windows.ps1
```

**Шаг 4 — health-check:**

```bash
./setup/health-check.sh     # Mac/Linux
.\setup\health-check.ps1    # Windows
```

Покажи результат. Если все 5 пунктов зелёные — готово.

**Шаг 5 — если что-то пошло не так:**
Предложи rollback:

```bash
./setup/rollback.sh         # Mac/Linux
.\setup\rollback.ps1        # Windows
```

Rollback восстанавливает папку из бэкапа, удаляет симлинк, убирает задачу auto-pull.

## Частые проблемы

- **Windows PowerShell 5.1 + кириллица в скрипте → парсер падает.** Решение: `.ps1` должен быть в UTF-8 с BOM. Без BOM PowerShell 5.1 читает как cp1251. Скрипты в этом ките уже с BOM.
- **`Register-ScheduledTask` падает на `[TimeSpan]::MaxValue`.** XML-схема Task Scheduler не принимает 10-значные дни. В ките используется `(New-TimeSpan -Days 3650)` — 10 лет.
- **Симлинк не работает в WSL1.** WSL2 поддерживает. В WSL1 — только обычная папка + ручной `git pull` в cron.

## Важно

- Не запускай скрипт без предупреждения если активная папка памяти не пустая — в ней могут быть локальные правки которых нет в репо. Скрипт их смержит (по `mtime`), но это хорошо проговорить вслух.
- Если Junction/симлинк уже корректный — скрипт выйдет с «ок», ничего не сломает.
- Путь к репо может отличаться — если не уверен, **спроси**.
